# Lessons

- Qt 交互界面中的控制按钮不能只做点击动作，必须同时提供持续的当前选中态反馈，让用户明确知道当前生效的舵令和车令。
- 当本轮工作已经围绕某份需求草稿或输入素材展开实现时，不要自行把这些工作区输入文件排除在提交之外；若用户未明确排除，应默认随同实现一起纳入提交。
- 讨论集成日志方案时，不要先入为主地把 `stdout` 当成默认协议；若 GUI 场景下标准输出会与诊断信息或宿主环境耦合，应优先采用明确目录和命名规则的文件输出方案。
- 当需求明确要求“形成一个新分支”或“维护两个分支”时，必须先检查并切换到目标分支后再开始修改，不能在 `main` 上直接落实现。
- 当用户要求保留现有配置入口或兼容旧分支的核心配置机制时，不能为了简化实现擅自改成代码内置默认值；应优先保留原有配置加载链路，再在其上做最小裁剪。
- 遇到时间累积器、状态机推进、资源生命周期这类逻辑稍复杂的函数时，要补充简洁注释，说明关键控制意图和保护条件，而不是只留裸代码。
- Qt 界面做视觉改造时，不能只满足“有主题色”；必须同时校准透明度、留白密度、按钮字号和默认态明度，避免出现过暗、过厚重的军事风样式，并且对 `QLabel` 这类简单文本控件优先保持默认样式，除非用户明确要求强化装饰。
- 当用户明确给出控件状态语义时，必须把默认态、按下态、选中态分别对齐到该语义，不能只改背景色而忽略字体颜色与层级关系；`QGroupBox` 标题还需要单独验证是否被边框或内容区挤压。
- 当用户确认当前界面已经符合要求时，不要继续推进高侵入性的窗口框架改造；像自绘标题栏这类会改变系统交互模型的方案，除非用户明确要求保留，否则应优先作为可选建议而不是直接落实现。
